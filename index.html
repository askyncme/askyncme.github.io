Omokai

status: Work in progress

Solo developer · Technical assignment

Omokai closes the loop from spoken intent to physical robot motion. A natural-language instruction is parsed by an LLM into a structured mission, checked against a deterministic safety validator, and executed as a real Nav2 navigation task inside Gazebo. The goal wasn't just to make an LLM output JSON, it was to build the guardrails that make it safe to act on.

KEY DECISIONS —Chose Groq (llama3-8b-8192) over local inference. Ollama technically worked but was unusably slow on CPU-only hardware. Groq's free tier gave the best latency-to-reliability tradeoff, the priority was a model that responds consistently. Put a hard validation layer between the LLM and the executor. The LLM is treated as untrusted input. Nothing reaches Nav2 without first passing schema checks and safety bounds, this was a deliberate design choice, not an afterthought, because an LLM hallucinating a bad waypoint should fail loudly before it ever reaches a motor command.

XLR8

status: voice pipeline complete, fine tuning drive system

Solo developer · Voice-driven robotics assistant

A modular voice-assistant system built on ROS 2, where independent nodes handle speech recognition, intent classification, LLM-based reasoning, and speech synthesis, communicating entirely through ROS topics, services, and actions. Designed as the interaction layer for a robotics platform, with a defined interface for downstream robot-action execution.

Key Decision — Implemented speech output as a cancellable ROS 2 action rather than a fire-and-forget service, enabling real-time barge-in: the system stops talking the instant the user starts speaking again. Allowed Robot Motion by voice commands through micro-ros interface.

Quadruped Robot

status: earlier project

Solo developer · personal project

A 12-DOF quadruped robot built from the ground up on an Arduino Nano and twelve SG90 servo motors, with coordinated gait control and motion sequencing for walking and expressive movements.

Key Decision — Selected a 12-DOF (3 DOF per leg) architecture to enable natural leg articulation while keeping the control system lightweight enough for an Arduino Nano. Developed predefined gait sequences with synchronized joint trajectories, prioritizing repeatable locomotion and smooth coordination over computationally intensive approaches such as inverse kinematics.

Multirotor UAV

status: All India Rank 11 in Phase-I

Electronics Head · SAE AeroTHON, Team Squadrone

Designed and integrated the electronics architecture for a multirotor UAV developed for the SAE AeroTHON competition. Responsible for selecting and integrating onboard hardware, including the flight controller, companion computer, sensors, power distribution, and communication interfaces, while collaborating with the mechanical and software teams to build an autonomous aerial platform. The UAV was designed to perform autonomous navigation, aerial surveying, object detection, and payload delivery as part of the competition mission.

Key Decision — Selected a Pixhawk-based flight control architecture paired with a Raspberry Pi companion computer, separating real-time flight control from high-level autonomy. This enabled reliable flight performance while providing sufficient computing capability for perception and mission logic. Chose system components by balancing payload capacity, flight endurance, and onboard processing requirements to meet the competition objectives.
